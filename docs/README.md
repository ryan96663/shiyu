## 为什么需要食遇？

### 独自用餐，是 9200 万独居人口每天面对的现实

你可能经历过这些：

- **一个人坐在餐厅里**，边吃饭边刷手机，偶尔抬头看看周围，继续低头——这不是享受美食，是"完成吃饭任务"
- **打开点评 App 想看看什么菜好吃**，刷了 10 分钟分不清哪些是真实评价、哪些是刷的、哪些是 KOL 收钱写的
- **排队等位半小时**，除了刷短视频无事可做，想提前了解店里情况但没有渠道
- **隔壁桌聊得挺热闹**，你想搭句话——"这个菜好吃吗？"——但想想还是算了

### 现有产品为什么解决不了？

| 你可能会用 | 问题 |
|-----------|------|
| **大众点评 / 小红书** | 水军泛滥、KOL 商业合作主导、你无法信任一个陌生人三周前写的评价 |
| **微信群** | 没门槛、没退出机制、成员身份无法验证、"谁都能进"意味着"谁都不会认真聊" |
| **Soul / 探探** | 场景错配——你只想问问这个菜好不好吃，对方可能在 20 公里外滑你 |
| **直接搭讪** | 压力大、怕被拒、怕尴尬——"万一对方不想理我呢" |

**核心矛盾**：你想和"正在同一家餐厅、看同一份菜单、吃同一个厨师做的菜"的人聊天——但现有的社交产品没有一个是围绕这个场景设计的。

## 食遇怎么解决？

### 核心理念：把餐厅变成"局域网"

每个餐厅是一个独立的社交域。**只有真实在店的消费者才能进入这个域**，与同店其他顾客互动——就像古早网吧，每台电脑只能和同网吧的其他电脑联机。

### 三重验证 · 确保"他真的是隔壁桌吃饭的人"

| 验证条件 | 实现方式 | 杜绝了什么 |
|---------|---------|-----------|
| ① GPS 定位在店内 | Haversine 公式计算距离 ≤ 50m，GPS 精度 ≤ 10m | 不在店的人 |
| ② 连接店家 WiFi | 匹配店铺 SSID + BSSID | GPS 漂移/伪造 |
| ③ 美团订单核销 | 订单 ID + 核销码双重验证 | 蹭 WiFi 的路人 |

**三个条件必须同时满足，缺一不可。** 这是整个产品信任体系的基石。

### 三种社交方式，三层边界感

```
┌────────────────────────────────────────────┐
│            食遇社交体系                      │
│                                            │
│  🗣 群聊大厅                                │
│  ├─ 在店顾客自动入群，匿名桌号显示           │
│  ├─ 禁止私聊、商家不可与顾客交互             │
│  ├─ 在店≤6小时权限，离店2小时自动退群        │
│  └─ 用途：菜品讨论、闲聊、找搭子             │
│                                            │
│  👋 打招呼                                  │
│  ├─ 一键随机向同店某顾客发送问候             │
│  ├─ 15条预设+30字自定义，不可指定目标        │
│  └─ 用途：破冰、问推荐、纯好奇              │
│                                            │
│  🫧 漂流瓶                                  │
│  ├─ 店外用户投放问题→随机分配给店内顾客      │
│  ├─ 3分钟超时入公共池，每日限10条            │
│  └─ 用途：等位提前了解、跨时空互动           │
└────────────────────────────────────────────┘
```

关键设计原则：**禁止私聊**。不是技术限制，是产品选择——让用户可以放心说话，不用担心被骚扰。

## 功能全景

### 店铺场景页（核心交互界面）

进入餐厅后看到一张完整的餐厅场景图——墙壁、窗户、灯光、地板、桌椅全部画在一张图里。点击任何一张桌子，弹出对话气泡（随机预制文案），模拟"隔壁桌在说话"的感觉。

同时，群聊消息以两种方式展示：
- **浮动聊天面板**：半透明层叠在场景底部，自动滚动到最新消息
- **弹幕**：在场景下方横向滚动，模拟"旁观群聊"的感觉

| 模块 | 做什么 | Demo 状态 |
|------|--------|----------|
| 🏠 **首页** | 店铺搜索（30 条 Mock + 实时过滤）、美食分类、热门推荐 | ✅ |
| 🎬 **餐厅场景** | 一体化背景图 + 热区气泡 + 自动播放群聊 Demo | ✅ |
| 💬 **群聊** | 匿名（桌号）聊天、弹幕、消息面板自动滚动 | ✅ |
| 👋 **打招呼** | 15 条预设 + 30 字自定义、随机匹配、不可指定 | ✅ |
| 🫧 **漂流瓶** | 扔/捞/回复、可见性控制、每日限额、超时公共池 | ✅ |
| ⭐ **AI 榜单** | 1-5 星评分、文字评价、AI 自动归类 + 情感分析 | ✅ |

## 技术架构

### 整体架构图

```
┌──────────────────────────────────────────────┐
│              前端 · 微信小程序                 │
│                                              │
│  原生框架 WXML/WXSS/JS                        │
│  v6.0 渲染：一体化背景图 + 百分比热区覆盖       │
│                                              │
│  核心理念：1 张完整场景图 + JS 坐标数组         │
│           → 设计师控制视觉，代码只负责放图       │
└──────────────────┬───────────────────────────┘
                   │  HTTP REST (wx.request)
                   │  WebSocket (适配中)
┌──────────────────▼───────────────────────────┐
│           后端 · Node.js API 服务              │
│                                              │
│  Express 4.x + Socket.io + node-cron          │
│  JWT 双 Token（access 2h + refresh 7d）       │
│  Winston 日志 + 内存存储降级模式               │
│                                              │
│  路由：auth / user / store / group /          │
│        bottle / review / ai                   │
│  服务：socketService · scheduler              │
│  工具：geoUtils(Haversine) · asyncHandler     │
└──────────────────────────────────────────────┘
```

### 渲染方案演进（为什么选 v6.0）

| 版本 | 方案 | 致命问题 |
|------|------|---------|
| v4.0 | Canvas 2D 等距引擎 + 精灵图 | 微信原生组件层级冲突、DrawCall 过多、Canvas API 受限 |
| v5.x | DOM 绝对定位 + 独立 PNG 桌位素材拼接 | 素材格式不匹配（JPG 改扩展名为 PNG）、加载 500 错误、坐标偏移 |
| **v6.0** | **单张一体化背景图 + 百分比坐标热区** | ✅ 一张图完成渲染，零拼接，零素材加载故障 |

**v6.0 的核心哲学**：从架构层面消除故障源。不再依赖多张素材的组合加载，不再依赖等距投影坐标转换。设计师画好场景图 → 代码层在桌上覆盖透明可点击热区 → 完成。

### 关键技术决策

| 决策 | 选型 | 理由 |
|------|------|------|
| 前端框架 | 微信原生 | 不增加构建复杂度，体积最小 |
| 场景渲染 | 单张背景图 + 百分比热区 | 零运行时拼接，消除素材加载类故障 |
| 热区坐标 | 百分比（left%/top%） | 适配不同屏幕宽度，无需重新计算 |
| 认证方案 | JWT 双 Token | access 2h 防泄露，refresh 7d 免频繁登录 |
| 数据存储 | 内存降级模式（Demo） | 零环境依赖，启动即用；生产环境切换 MongoDB |
| Demo 脚本 | setTimeout 批量注册 | 按秒数偏移精确触发，无需轮询 |

## 快速开始

### 环境

| 组件 | 要求 |
|------|------|
| Node.js | ≥ 18.0.0 |
| 微信开发者工具 | 最新稳定版 |
| MongoDB / Redis | 可选（Demo 阶段内存存储） |

### 后端

```bash
cd backend
npm install
node src/app.js
# → 服务器启动成功，监听端口: 3000
```

### 前端

1. 微信开发者工具 → 导入项目 → 选择 `frontend/` 目录
2. AppID 选**测试号**
3. 设置 → **不校验合法域名** ✓
4. 编译

### 终端测试

```bash
# Mock 登录
curl -X POST http://localhost:3000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"code":"test_123"}'

# 搜索店铺
curl "http://localhost:3000/api/v1/store/search?latitude=39.9042&longitude=116.4074"
```

## Demo 场景演示

进入店铺详情页后，系统自动播放 **5 分钟完整社交场景**：

### 场景 A · 菜品点评（群聊）

7 位"顾客"在群聊中围绕"水煮牛肉到底好不好吃"热烈讨论——推荐、争论、实测打分、隐藏菜单分享。过程中 10 号桌新人邀请 2 号桌心情不好的大学生过来吃小酥肉 → 最后默默帮对方结了账 → 加微信。

**展示效果**：
- 群聊消息按真实节奏逐条弹出（每 5 秒一句，共 62 条）
- 场景图上对应桌位弹出对话气泡，3 秒消失
- 每张桌子累计发言数常驻显示
- 浮动聊天面板半透明叠在场景底部

### 场景 B · 旅游问路（打招呼）

4 号桌游客通过"打招呼"随机匹配到 11 号桌北京本地人，问"故宫怎么走"→ 对方详细指路 + 推荐角楼拍照 +"北京人嘴贫心热"→ 感谢收尾。

**展示效果**：
- Toast 通知弹出（"4号桌向你打招呼"）
- 对话内容同步进入群聊面板和弹幕

### 两条线在 03:00~03:55 时段重叠

群聊温暖收尾 + 打招呼路线建议并行进行，展示产品两种社交形态如何自然共存。
