# 美食社交小程序 - 产品需求文档(PRD)

## 1. 项目概述

### 1.1 项目名称
美食社交小程序 - "食遇"

### 1.2 项目背景
美团致力于拓展美食社交新增量板块，通过将游戏与现实场景深度融合，创造独特的线上线下社交体验。消费者在餐厅场景中既作为真实顾客存在，又作为虚拟NPC与其他消费者进行互动，实现虚实结合的社交创新。

### 1.3 核心理念
游戏化社交 + 真实性保障 + 边界感维护 + 实时互动

## 2. 产品愿景

创造基于真实餐饮场景的即时社交小程序，让用户在享受美食的同时，能够与同店消费者进行安全、有趣的互动，增强用餐体验，同时为商家带来更多流量和用户粘性。

## 3. 目标用户

### 3.1 主要用户群
- 年轻消费者(18-35岁)
- 美食爱好者
- 社交型用餐者
- 探店达人

### 3.2 用户痛点
- 想了解同店消费者的真实评价
- 想与同好分享美食体验
- 排队等待时缺乏互动
- 补充本地生活电商领域社交空白

## 4. 核心功能模块

### 4.1 店家实时群

#### 4.1.1 功能描述
仅到店用户可参与的临时群聊，保证评价真实性和实时性。

#### 4.1.2 触发条件
三项条件必须**同时满足**：
1. **到店定位**：GPS定位在店内范围内（精度≤10米）
2. **店家WiFi连接**：成功连接店铺专属WiFi
3. **美团核销**：通过美团完成订单核销/点单/优惠券买单

#### 4.1.3 功能特性

**进入机制**：
- 满足触发条件后自动进入群聊界面
- 发言昵称以桌号形式显示（如"5号桌"）
- 可选择匿名状态

**群成员**：
- 当前在店消费者
- 满足进入条件后6小时内的消费者
- 实时在线状态管理

**交流方式**：
- 菜品评价与推荐
- 环境分享与建议
- 美食交友
- 随机话题讨论

#### 4.1.4 约束机制

**时间限制**：
- 在店定位：最多6小时群聊权限
- 离开店铺定位范围后：2小时内自动退群
- 总时长不超过6小时
- 主动退群功能

**安全约束**：
- 不支持私聊（保障用户隐私）
- 店外人员无法参与讨论
- 短时间重复进入限制（防刷屏）

**旁观模式**：
- 店外人员可查看群聊内容（弹幕形式）
- 但不能参与讨论

#### 4.1.5 应用示例
类似于网吧网吧域模式：
- 每个网吧是一个独立域
- 每台电脑只能与同网吧内其他电脑联机
- 用户仅限网吧内，不与其他网吧交叉连接

### 4.2 餐厅场景展示系统（一体化背景图方案）

#### 4.2.1 设计理念

放弃之前”背景图 + 独立桌位素材分层拼接”的复杂架构，改为**一张完整场景图 + 透明热区覆盖**的极简方案。桌椅、墙壁、地板、窗户等元素由设计师统一绘制在一张完整背景图中，代码层仅负责”放图 + 在桌子上覆盖可点击的透明热区”。

**方案优势**：
- 彻底解决素材加载失败、格式不匹配、坐标偏移等渲染问题
- 一张图即可完成场景展示，无需维护多张桌位素材
- 代码量大幅减少，不再需要等距坐标计算、变体分发、深度排序等复杂逻辑
- 素材由设计师全权控制，迭代只需替换一张图

#### 4.2.2 渲染架构

**2 层结构**：

| 层级 | 内容 | 实现方式 |
|------|------|---------|
| Layer 1: 场景背景图 | 完整餐厅场景（桌椅已画在图中） | `<image>` 组件，`mode=”widthFix”` 全宽适配 |
| Layer 2: 热区覆盖层 | 透明可点击区域，覆盖在每张桌子的位置 | 绝对定位 `<view>`，`bindtap` 响应点击 |

**热区坐标规范**：
- 使用**百分比坐标**（left%、top%），确保不同屏幕宽度下热区与背景图桌位对齐
- 热区尺寸：60×60px ~ 80×80px，确保手指点击友好
- 坐标配置以数组维护在页面 JS 中，便于调整

```javascript
// 热区配置示例
hotspots: [
  { id: 1, left: '15%', top: '42%', label: '1号桌' },
  { id: 2, left: '40%', top: '42%', label: '2号桌' },
  { id: 3, left: '65%', top: '42%', label: '3号桌' },
  { id: 4, left: '27%', top: '62%', label: '4号桌' },
  { id: 5, left: '53%', top: '62%', label: '5号桌' },
]
```

#### 4.2.3 交互设计

**桌位点击（对话气泡）**：
- 点击桌位热区 → 该桌子上方弹出对话气泡
- 气泡内容从预制文案池（8-10条）中随机选取
- 气泡 3 秒后自动消失（CSS animation 淡入淡出）
- 同时 Toast 显示桌号（如”3号桌”）
- Demo 阶段不做”有人坐/空桌”状态区分

**预制文案池**：
- “这家招牌菜绝了！”
- “强烈推荐这个，每次来都点”
- “今天人好多啊”
- “味道一如既往的好”
- “第一次来，有推荐的吗？”
- “隔壁桌的看着也好吃”
- “服务态度真不错”
- “环境很舒适，适合聚餐”
- “份量很足，性价比高”
- “排队排了好久，但值得”

**前台点击**：
- 点击前台区域 → Demo 模式 Toast 提示
- 生产模式 `wx.navigateToMiniProgram` 跳转美团点餐

**打招呼（保留现有功能）**：
- 店内消费者可随机向同店在线顾客发送问候
- 通过下方互动栏”打招呼”按钮触发，跳转独立页面

**漂流瓶交互（保留现有功能）**：
- 店外用户投放 → 店内用户拾取 → 回复
- 通过下方互动栏”丢漂流瓶”/”捡漂流瓶”按钮触发
### 4.3 星级榜单系统

#### 4.3.1 评分机制
- 仅满足触发条件的消费者可参与评分
- 评分范围：1-5星
- 必须附加文字评价或推荐理由

#### 4.3.2 榜单展示
- 实时更新的星级排行
- 菜品维度评分
- 综合评价展示
- 评价内容可查看

#### 4.3.3 排序算法
- 加权平均评分
- 时间衰减因子
- 评价质量权重

### 4.4 打招呼功能

#### 4.4.1 功能描述
店内消费者可随机向其他在线顾客发送问候

#### 4.4.2 使用条件
- 仅限店内实时消费者
- 必须满足三项触发条件

#### 4.4.3 功能特性
- 随机匹配对象
- 不支持指定目标
- 保护用户隐私
- 一键打招呼

#### 4.4.4 界面位置
- 位于界面右上角
- 仅对符合条件的用户显示

### 4.5 漂流瓶系统

#### 4.5.1 功能概述
允许店外用户对店内消费者进行提问和互动

#### 4.5.2 投放机制

**适用人群**：
- 店外对店铺感兴趣的消费者
- 排队等待的顾客

**投放方式**：
- 点击"丢个漂流瓶问问"按钮
- 撰写问题内容
- 选择可见性模式：
  - 模式1：仅拾取者可见
  - 模式2：全部人可见

#### 4.5.3 收取与响应

**分发机制**：
- 随机派发给店内在线顾客
- 3分钟响应时限

**处理流程**：
- 3分钟内未响应：转入公共漂流瓶池
- 响应成功：建立对话连接

**查看权限**：
- 仅拾取者可见：仅指定接收者可查看和回复
- 全部人可见：群内所有用户可查看和回复

#### 4.5.4 界面显示

**店外用户界面**：
- 右上角"丢个漂流瓶问问"按钮

**店内用户界面**：
- 右上角"漂流瓶的桶"按钮
- 显示可查看的所有漂流瓶
- 新漂流瓶消息推送提醒

#### 4.5.5 限制规则

**数量限制**：
- 每日最多10个问题
- 每日最多5个不同店铺
- 单个漂流瓶对话上限：20条消息

### 4.6 实时弹幕系统

#### 4.6.1 展示位置
- 店铺图片左下角
- 实时滚动展示

#### 4.6.2 内容来源
- 店家实时群聊消息
- 过滤敏感内容
- 匿名显示

#### 4.6.3 显示规则
- 滚动速度自适应
- 支持关闭/开启
- 24小时历史记录

### 4.7 场景更新策略

#### 4.7.1 Demo 阶段
- 使用 1 套通用餐厅场景背景图
- 不区分时间/天气/餐厅类型
- 背景图由设计师统一绘制，迭代时直接替换图片文件
- 热区坐标在 JS 配置中调整，不涉及前端架构变更

#### 4.7.2 后续演进方向
- 按餐厅类型（川菜/粤菜/日料/烧烤）提供 4 套背景图
- 按时间（白天/夜间）+ 天气（晴天/雨天）烘焙变体
- 前端根据业务状态机条件切换背景图 src，无需运行时渲染

## 6. 技术架构

### 6.1 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                  前端（微信小程序）                        │
│  原生框架 + WXML/WXSS/JS                                 │
│  核心：一体化背景图 + 热区覆盖 + 对话气泡 + 弹幕            │
│  UI 组件库：Vant Weapp（部分使用）                        │
└────────────────────┬────────────────────────────────────┘
                     │ HTTP REST (wx.request)
                     │ WebSocket (wx.connectSocket)
┌────────────────────▼────────────────────────────────────┐
│                  后端 API 服务（Node.js）                  │
│  Express + Socket.io + node-cron + Winston               │
│  JWT 认证（access token 2h + refresh token 7d）          │
│  数据存储：MongoDB（持久化）+ Redis（缓存/会话）           │
│        ⚠️ 当前为内存 fallback，未正式接入                  │
│  核心模块：auth / user / store / group / bottle / review  │
└─────────────────────────────────────────────────────────┘

素材层（一体化方案）：
  1 张完整餐厅场景图（桌椅墙壁地板统一绘制）
  + JS 热区坐标配置数组
  → 不再需要独立桌位 PNG 素材、等距坐标引擎、Canvas 渲染
```

### 6.2 性能指标体系

#### 渲染性能要求
- **场景加载**：背景图加载完成时间 ≤ 2 秒（单张图片，无拼接开销）
- **启动性能**：小程序冷启动时间 ≤ 3 秒
- **内存控制**：页面内存峰值 < 200MB（无 Canvas 离屏渲染缓冲区）

#### 实时交互性能
- **消息延迟**：WebSocket 消息传输延迟 ≤ 200ms
- **交互响应**：点击热区到气泡展示 ≤ 100ms
- **动画性能**：对话气泡 CSS animation 60fps 流畅运行

#### 设备兼容性
- **低端机适配**：iPhone SE / Android 中端机保底流畅
- **网络容错**：弱网环境下保持基本功能可用，背景图加载失败显示占位图
- **微信开发者工具**：确保图片路径正确、格式匹配，避免 500 错误

### 6.2 核心服务

**用户服务**：用户认证、权限管理、位置验证
**实时聊天服务**：WebSocket连接、消息分发、群聊管理
**场景实时渲染**：店铺实际情况确定、素材选择策略、场景渲染、聊天显示功能
**漂流瓶服务**：消息分发、状态管理、权限控制
**排行榜服务**：评分计算、榜单维护、数据更新

## 8. 商业模式与价值

### 8.1 商业价值

**对消费者**：
- 打破独自用餐的社交隔离，以匿名、安全的方式与同店顾客互动
- 获取真实消费者的一手菜品评价，避免水军和虚假评论误导
- 等位时可提前了解店内氛围、人气菜品，提升决策效率
- 游戏化场景让用餐体验更有趣味性，增加到店消费动力

**对商家**：
- 真实消费者产生的 UGC 内容自带公信力，低成本口碑传播
- 场景化社交增强用户粘性，提升复购率和客单价
- 实时反馈帮助商家了解菜品满意度，指导菜单优化
- 店铺在平台的曝光度提升，吸引更多潜在顾客到店

**对美团**：
- 拓展本地生活社交新板块，与现有外卖/到店业务形成协同
- 获取用户真实消费场景的行为数据，丰富用户画像
- 差异化竞争优势——目前无同类产品在"店内社交"领域深耕
- 增强平台生态竞争力，从工具型平台向社交型平台延伸

### 8.2 可行性分析

**技术可行性**：
- 前端采用微信小程序原生框架，无重型第三方依赖
- 一体化背景图方案大幅降低渲染复杂度，减少故障点
- 后端 Node.js + Express 技术栈成熟稳定
- 实时通信采用 WebSocket（当前有待兼容性完善）
- AI 能力可渐进式接入（当前 Demo 阶段用预制文案替代）

**市场可行性**：
- 年轻用户（18-35 岁）对社交+游戏化产品接受度高
- 餐饮社交是尚未被充分开发的蓝海市场
- 美团生态内有天然的流量入口和使用场景
- 差异化竞争优势明显——竞品多为独立社交 App，缺乏场景绑定

**运营可行性**：
- 借助美团现有商家资源和用户基础，冷启动成本低
- 三重验证（GPS+WiFi+核销）天然绑定美团订单体系，防作弊
- 匿名+禁止私聊的设计降低运营审核压力
- 先 Demo 验证核心体验，再逐步接入真实商家数据
