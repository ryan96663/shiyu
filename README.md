# 美食社交小程序 - 产品需求文档(PRD)

## 1. 项目概述

### 1.1 项目名称
美食社交小程序 - "食遇"

### 1.2 项目背景
美团致力于拓展美食社交新增量板块，通过将游戏与现实场景深度融合，创造独特的线上线下社交体验。消费者在餐厅场景中既作为真实顾客存在，又作为虚拟NPC与其他消费者进行互动，实现虚实结合的社交创新。

### 1.3 核心理念
游戏化社交 + 真实性保障 + 边界感维护 + 实时互动

## 2. 产品愿景

创造基于真实餐饮场景的即时社交小程序，让用户在享受美食的同时，能够与同店消费者进行安全、有趣的互动，增强用餐体验，同时为商家带来更多流量和用户粘性。

## 3. 目标用户

### 3.1 主要用户群
- 年轻消费者(18-35岁)
- 美食爱好者
- 社交型用餐者
- 探店达人

### 3.2 用户痛点
- 想了解同店消费者的真实评价
- 想与同好分享美食体验
- 排队等待时缺乏互动
- 补充本地生活电商领域社交空白

## 4. 核心功能模块

### 4.1 店家实时群

#### 4.1.1 功能描述
仅到店用户可参与的临时群聊，保证评价真实性和实时性。

#### 4.1.2 触发条件
三项条件必须**同时满足**：
1. **到店定位**：GPS定位在店内范围内（精度≤10米）
2. **店家WiFi连接**：成功连接店铺专属WiFi
3. **美团核销**：通过美团完成订单核销/点单/优惠券买单

#### 4.1.3 功能特性

**进入机制**：
- 满足触发条件后自动进入群聊界面
- 发言昵称以桌号形式显示（如"5号桌"）
- 可选择匿名状态

**群成员**：
- 当前在店消费者
- 满足进入条件后6小时内的消费者
- 实时在线状态管理

**交流方式**：
- 菜品评价与推荐
- 环境分享与建议
- 美食交友
- 随机话题讨论

#### 4.1.4 约束机制

**时间限制**：
- 在店定位：最多6小时群聊权限
- 离开店铺定位范围后：2小时内自动退群
- 总时长不超过6小时
- 主动退群功能

**安全约束**：
- 不支持私聊（保障用户隐私）
- 店外人员无法参与讨论
- 短时间重复进入限制（防刷屏）

**旁观模式**：
- 店外人员可查看群聊内容（弹幕形式）
- 但不能参与讨论

#### 4.1.5 应用示例
类似于网吧网吧域模式：
- 每个网吧是一个独立域
- 每台电脑只能与同网吧内其他电脑联机
- 用户仅限网吧内，不与其他网吧交叉连接

## 4.2 店铺场景生成系统

#### 4.2.1 前端像素风店铺场景渲染

**渲染架构**
- **技术方案**：采用 WXML 原生 DOM 节点 + Absolute 绝对定位方案（Demo 极速版）
  - Layer 1: 场景背景 - 店铺底图（512px × 512px 正方形，CSS background-image）
  - Layer 2: 桌位物件 - 80px × 80px 透明 PNG 素材绝对定位叠加
- **坐标体系**：全站统一 8×8 逻辑网格协议（Col/Row 范围 0-7），通过等距转换公式计算屏幕坐标
- **深度排序**：利用 CSS z-index: (row * 100 + col) 实现 2.5D Y-Sort 深度遮挡剪裁

**素材规范**
- **背景素材**：1张512×512px场景主大底图（含天气烘焙变体）
- **桌位素材**：7张80×80px透明底PNG标准餐桌素材
  - table_empty.png（空桌）
  - table-2persons-v1/2/3.png（2人桌变体）
  - table-4persons-v1/2/3.png（4人桌变体）
- **路径规范**：/images/restaurant/ 目录下统一管理
- **素材定位**：锚点统一锁定在图片底部正中心（transform: translate(-50%, -100%)）

**渲染原则**
- **尺寸固定**：桌位素材在 WXSS 中固定为 117rpx × 117rpx
- **像素保真**：image-rendering: pixelated 保持像素风格锐利
- **坐标转换**：2.5D 等距投影公式 screenX = (col - row) * 32, screenY = (col + row) * 16
- **变体分发**：采用余数分发算法（tableIndex % variants.length）自动交错渲染多款美术变体

**动态场景渲染**
- **背景切换**：静态底图切换（Baking 烘焙法），根据环境参数直接切换大图 src
- **桌位状态**：根据实时 occupancy 数据动态切换对应素材
- **布局计算**：JS 内部计算等距坐标，转换为绝对定位 left/top 值
- **深度排序**：CSS z-index 实现，数值越大越靠前显示

**瓦片地图架构**（Canvas备用方案）
- **三层架构**：
  - Layer 1: 地面瓦片层 - 餐厅地表结构基础网格
  - Layer 2: 物件层 - 桌椅、NPC、前台资产（实施Y轴深度排序）
  - Layer 3: UI层 - 对话气泡、数字标签、交互面板
- **动态伸缩**：支持纵向网格扩充，Head+Middle+Tail结构
- **碰撞检测**：基于瓦片网格的锚定和Hit Test

**性能优化**
- **视口裁剪**：只渲染当前可视范围内的物件
- **素材缓存**：预加载+LRU缓存策略
- **内存控制**：iOS低端机内存峰值 < 1GB
- **渲染兼容性**：在微信开发者工具环境下确保素材兜底（空路径处理）


**输入参数**：（需更改）
- 店铺画布原始图片
- 实时餐桌数量
- 桌椅与小人组合图片
- 店铺评价数据
- 群聊内容氛围分析
- 当前天气情况
- 昼夜时间

**环境状态系统**

**场景状态表**：
| 场景状态 | 色彩滤镜 (Color Tint) | 粒子特效 |
| :--- | :--- | :--- |
| **白晴** | 原色 | 无 |
| **白雨** | 冷蓝色调偏移 | 像素雨丝粒子层 (60°) |
| **黑晴** | 压暗/蓝紫色调 | NPC/桌椅上方像素光晕 |
| **黑雨** | 极暗色调 + 蓝紫偏移 | 像素雨丝 + 夜间像素光晕 |

**实现机制**：
- **色彩调节**：通过Shader或Alpha遮罩动态调节画面色调
- **天气响应**：基于实时天气API数据自动切换状态
- **时间响应**：根据昼夜时间自动匹配对应场景状态
- **设备兼容性**：确保在微信小程序环境下保持视觉一致性

**输出特性**：（需更改）
- 像素风格餐厅图片
- 餐桌数量严格对应真实座位
- 仅考虑餐厅有人用餐的桌子数量，并将此处空桌椅替换成有小人的桌椅组合素材
- **环境状态同步**：背景图按时间×天气组合自动切换

#### 4.2.2 动态店内顾客显示

**在位顾客**：
- 有人落座的餐桌实时替换场景内空桌椅为有人落座的桌椅小人组合
- 顾客形象为匿名NPC形式
- 不显示具体个人信息

**NPC对话系统**

**数据结构**：
```typescript
{
  total_chats: number,      // 长期显示计数
  speech_history: string[], // 存储全部对话
  timer: number            // 气泡消失倒计时 (120s)
}
```

**即时对话气泡**：
- **渲染位置**：NPC头顶 ΔY 处
- **渲染技术**：使用9-Slice框渲染像素气泡对话框
- **生命周期**：120秒倒计时（7200帧@60fps）
- **更新机制**：新消息到达时覆盖原有文本并重置计时器
- **视觉规范**：确保对话框边框像素点不因拉伸而变形

**长效发言计数器**：
- **显示形式**：常驻阿拉伯数字标签
- **渲染位置**：NPC精灵正上方（独立于气泡，优先级极高）
- **数据绑定**：强绑定 current_total_chats 数值
- **显示规则**：只要在生存周期内发过言，标签将持续显示

**历史对话面板**：
- **触发方式**：点击阿拉伯数字标签触发手势劫持
- **交互阻断**：成功点击时阻断底层Tilemap的拖拽和滑动操作
- **面板结构**：独立于Canvas之上的原生 ScrollView 组件
- **内容布局**：
  - 分条呈现历史对话记录
  - 每条记录包含【发言时间】+【台词文本】
  - 支持纵向滑动查看完整历史
- **数据来源**：遍历 speech_history 数组分条渲染

#### 4.2.3 交互设计

**与商家交互**：
- 点击前台区域可选择跳转点餐小程序或由商家推荐菜品（新更新功能）
- 商家账户禁止与普通消费者互动
- 与前台交互功能设计

**店内顾客与店内顾客交互**
- 打招呼功能支持在店内随机向在线顾客提问

**店外顾客与店内顾客交互**
- 漂流瓶可以随机被投放至店内顾客，超时未被点开会流进店内漂流瓶公海，可以被其他人重新拾取查看并回复
- 即时对话气泡
	- 新台词实时压入 `history_speech_list` 数组，且 `current_total_chats += 1`
	- **渲染表现**：在 NPC 头顶上方固定偏移 $\Delta Y$ 处，使用像素九切片（9-Slice）技术渲染一个气泡框（确保对话框边框的像素点不因拉伸而变形）。
	- 2mins 逻辑：气泡激活时，设置 `bubble_remaining_frames = 120 * FPS`（在 60 帧环境下为 7200 帧）。每帧递减。
		- 当计数器归零时，即时气泡置为隐藏（`bubble_active = false`）。
	    - 在中途 2 分钟内若触发了新台词，直接覆盖原有文本，并将帧数计数器重新重置为 7200 帧。
- 长效阿拉伯数字发言标签
	- 渲染规范：在 NPC 精灵正上方（独立于即时气泡，优先级极高），常驻渲染一个由像素框包裹的阿拉伯数字，数据源强绑定 `current_total_chats`。
	- 常驻逻辑：该数字标签不随 2 分钟即时气泡的隐藏而隐藏。只要该 NPC 在本次生存周期内讲过话，该数字标签将持续暴露。
- 历史对话详情面板（分条呈现交互）
	- 手势劫持（Hit Test）：当用户在手机屏幕上点击某个 NPC 头顶的“阿拉伯数字标签”时，前端判定射线成功击中热区，立即阻断并拦截底层 Tilemap 的拖拽与滑动滑块手势
	- 面板呈现机制
		- 弹出一个独立于游戏 Canvas 渲染层之上的底层小程序原生 `ScrollView` 组件面板（或高层级 DOM 视图层）。
		- 读取该 NPC 的 `history_speech_list` 数据队列。
		- **分条呈现布局**：遍历队列，每一条历史文本使用独立的像素风列表条目（Item 行）自上而下排列，必须包含【发言时间】与【台词文本】，支持纵向滑动回溯。

### 4.5 漂流瓶系统

#### 4.5.1 功能概述
允许店外用户对店内消费者进行提问和互动

#### 4.5.2 投放机制

**适用人群**：
- 店外对店铺感兴趣的消费者
- 排队等待的顾客

**投放方式**：
- 点击"丢个漂流瓶问问"按钮
- 撰写问题内容
- 选择可见性模式：
  - 模式1：仅拾取者可见
  - 模式2：全部人可见

#### 4.5.3 收取与响应

**分发机制**：
- 随机派发给店内在线顾客
- 3分钟响应时限

**处理流程**：
- 3分钟内未响应：转入公共漂流瓶池
- 响应成功：建立对话连接

**查看权限**：
- 仅拾取者可见：仅指定接收者可查看和回复
- 全部人可见：群内所有用户可查看和回复

#### 4.5.4 界面显示

**店外用户界面**：
- 右上角"丢个漂流瓶问问"按钮

**店内用户界面**：
- 右上角"漂流瓶的桶"按钮
- 显示可查看的所有漂流瓶
- 新漂流瓶消息推送提醒

#### 4.5.5 限制规则

**数量限制**：
- 每日最多10个问题
- 每日最多5个不同店铺
- 单个漂流瓶对话上限：20条消息

- 不再进行运行时 Canvas 绘图

### 6.2 性能指标体系

#### 渲染性能要求
- **帧率标准**：Canvas 2D 渲染维持 60fps（如适用）
- **内存控制**：iOS 低端机内存峰值 < 1GB
- **渲染管线**：DrawCall 控制在 10 以内
- **启动性能**：小程序冷启动时间 ≤ 3秒
- **场景加载**：餐厅场景渲染完成时间 ≤ 1.5秒

#### 实时交互性能
- **消息延迟**：WebSocket 消息传输延迟 ≤ 200ms
- **状态同步**：NPC状态更新响应时间 ≤ 500ms
- **群聊响应**：多用户同时发言时界面流畅度保障
- **动画性能**：对话气泡动画60fps流畅运行

#### 防掉帧策略
- **视口裁剪（Frustum Culling）**：仅绘制当前可视范围内的Tile
- **图层分级渲染**：按距离动态调整渲染细节层次
- **批量渲染优化**：sprite sheet 合并减少draw call
- **异步加载**：素材预加载避免运行时卡顿

#### 设备兼容性
- **低端机适配**：iPhone SE/Android 中端机保底流畅
- **内存兜底**：超出内存限制时自动降低画质
- **网络容错**：弱网环境下保持基本功能可用

### 6.2 核心服务

**用户服务**：用户认证、权限管理、位置验证
**实时聊天服务**：WebSocket连接、消息分发、群聊管理
**场景实时渲染**：店铺实际情况确定、素材选择策略、场景渲染、聊天显示功能
**漂流瓶服务**：消息分发、状态管理、权限控制
**排行榜服务**：评分计算、榜单维护、数据更新
